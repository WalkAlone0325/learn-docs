## 1 环境与工具链

### 1.1 为什么要学习 Docker？

开发中最头疼的问题之一就是"我机器上能跑"。不同开发者的系统环境不同，依赖版本不同，导致代码部署时问题层出不穷。Docker 将应用及其依赖打包成标准化的镜像，在任何安装了 Docker 的机器上都能以相同方式运行，彻底解决环境一致性问题。

### 1.2 安装 Docker Desktop / Docker Engine

Docker 有两种运行方式：**Docker Desktop**（含 GUI，推荐 macOS / Windows 使用）和 **Docker Engine**（纯 CLI，推荐 Linux 服务器使用）。

**macOS 安装：**

```bash
# 使用 Homebrew 安装 Docker Desktop
brew install --cask docker

# 安装后启动 Docker Desktop
open /Applications/Docker.app
```

**Linux 安装：**

```bash
# Ubuntu / Debian
sudo apt-get update
sudo apt-get install docker-ce docker-ce-cli containerd.io docker-compose-plugin

# 将当前用户加入 docker 组（避免每次使用 sudo）
sudo usermod -aG docker $USER
newgrp docker
```

**验证安装：**

```bash
docker --version
# 输出示例: Docker version 28.5.2, build abc1234

docker compose version
# 输出示例: Docker Compose version v2.40.3-desktop.1

docker info
# 查看 Docker 引擎详细信息，包括容器数、镜像数、存储驱动等
```

### 1.3 Docker CLI 基本命令

```bash
# 查看所有命令
docker --help

# 查看子命令帮助
docker run --help

# 查看系统信息
docker system df
# 查看磁盘使用情况（镜像、容器、卷、构建缓存）

docker system prune
# 清理未使用的资源（容器、网络、悬空镜像、构建缓存）
# 加上 -a 会清理所有未使用的镜像
```

### 1.4 验证端口未被占用

```bash
# 检查端口是否被占用
lsof -i :5432
# 无输出说明 5432 端口空闲

lsof -i :6379
# 无输出说明 6379 端口空闲
```

⚠️ **提示**：如果端口被占用，可以使用 `kill` 终止占用进程，或调整 Docker 映射端口。

### 1.5 开发环境准备

创建项目的目录结构：

```bash
mkdir -p ~/projects/docker-demo && cd ~/projects/docker-demo
mkdir src tests
```

初始化 Python 项目：

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install fastapi uvicorn[standard] sqlalchemy asyncpg redis alembic pydantic-settings
```

创建基础配置文件：

```bash
touch src/__init__.py
touch src/main.py
touch .env .dockerignore docker-compose.yml Dockerfile
```

💡 **提示**：将 `.venv` 加入 `.gitignore` 和 `.dockerignore`，避免将虚拟环境打包进镜像，导致镜像体积膨胀。

## 2 容器基础

### 2.1 为什么需要容器？

传统虚拟机（VM）需要完整的客户操作系统，启动慢、资源占用大。容器共享宿主机的操作系统内核，启动是秒级的，资源开销远小于虚拟机。容器让你可以在一台机器上运行多个隔离的应用实例，而不会产生 VM 那样的资源浪费。

### 2.2 容器 vs 虚拟机

| 特性 | 容器 | 虚拟机 |
|------|------|--------|
| 启动速度 | 毫秒~秒级 | 分钟级 |
| 大小 | MB 级 | GB 级 |
| 内核 | 共享宿主机内核 | 独立内核 |
| 隔离性 | 进程级（namespaces） | 完全隔离 |
| 资源占用 | 仅应用+依赖 | 完整 OS |
| 密度 | 一台机器可运行数百个 | 通常数十个 |
| 移植性 | 镜像跨平台一致运行 | 取决于 Hypervisor |

### 2.3 镜像与容器的关系

- **镜像**（Image）是只读模板，包含运行应用所需的代码、运行时、库、环境变量和配置文件
- **容器**（Container）是镜像的运行实例，可读可写
- 类比：镜像是 **类（Class）**，容器是 **实例（Instance）**

```bash
# 拉取镜像
docker pull python:3.14.5-slim

# 列出本地镜像
docker images

# 基于镜像创建并运行容器
docker run python:3.14.5-slim python --version
```

### 2.4 容器生命周期

容器状态转换：`created → running → paused → stopped → deleted`

```bash
# 1. 创建容器（但不启动）
docker create --name my-python python:3.14.5-slim

# 查看状态：created
docker ps -a

# 2. 启动容器
docker start my-python

# 3. 暂停容器（冻结进程）
docker pause my-python

# 恢复暂停
docker unpause my-python

# 4. 停止容器（发送 SIGTERM，等待后 SIGKILL）
docker stop my-python

# 5. 启动已停止的容器
docker start my-python

# 6. 删除容器（必须已停止，或用 -f 强制删除运行中的容器）
docker rm my-python
```

💡 **提示**：`docker stop` 会发送 SIGTERM 信号，给进程 10 秒的优雅关闭时间。超时后发送 SIGKILL。如果应用需要更长的关闭时间，使用 `docker stop -t 30 <container>` 指定超时秒数。

### 2.5 核心命令详解

```bash
# 运行容器（前台模式）
docker run nginx:latest

# 分离模式（后台运行）
docker run -d --name my-nginx -p 8080:80 nginx:latest

# 列出运行中的容器
docker ps

# 列出所有容器（包括已停止）
docker ps -a

# 停止容器
docker stop my-nginx

# 强制停止
docker kill my-nginx

# 删除容器
docker rm my-nginx

# 强制删除运行中的容器
docker rm -f my-nginx

# 进入运行中容器（交互模式）
docker exec -it my-nginx sh

# 查看容器日志
docker logs my-nginx

# 实时跟踪日志
docker logs -f my-nginx

# 查看容器内部进程
docker top my-nginx

# 查看容器资源使用
docker stats my-nginx
```

### 2.6 交互模式与分离模式

```bash
# 交互模式：-it（-i 保持 STDIN 打开，-t 分配伪终端）
docker run -it ubuntu:latest bash
# 进入容器后可以执行命令，exit 退出后容器停止

# 分离模式：-d（后台运行）
docker run -d --name web nginx:latest

# 分离模式中进入容器交互
docker exec -it web bash

# 分离模式中执行单条命令
docker exec web ls /etc/nginx
```

💡 **提示**：初学者常混淆 `docker run -it` 和 `docker exec -it`。`run` 创建新容器，`exec` 在已有容器中执行命令。

### 2.7 端口映射

```bash
# 随机映射宿主机端口到容器 80 端口
docker run -d -P nginx:latest

# 指定映射
docker run -d -p 8080:80 nginx:latest
# 宿主机 8080 → 容器 80

# 多端口映射
docker run -d -p 8080:80 -p 8443:443 nginx:latest

# 指定 IP 地址映射
docker run -d -p 127.0.0.1:8080:80 nginx:latest
# 只能通过 localhost 访问
```

**验证端口映射：**

```bash
docker run -d --name web -p 8080:80 nginx:latest
curl http://localhost:8080
# 输出 Nginx 默认欢迎页 HTML

docker stop web && docker rm web
```

### 2.8 对比：错误 vs 正确

❌ **错误方式**：在容器内 hardcode 端口

```bash
docker run -d python:3.14.5-slim python -m http.server 8000
# 这会在容器内启动 8000，但宿主机无法访问
```

✅ **正确方式**：通过 -p 暴露端口

```bash
docker run -d -p 8000:8000 python:3.14.5-slim python -m http.server 8000
# curl http://localhost:8000 即可访问
```

## 3 Dockerfile 基础

### 3.1 为什么需要 Dockerfile？

虽然可以直接从现有镜像启动容器，但这无法定制应用。Dockerfile 是构建自定义镜像的"配方"，告诉 Docker 如何一步步组装出你的应用镜像。

### 3.2 FROM 指令与基础镜像选择

`FROM` 指定构建的基础镜像。选择基础镜像时要权衡体积、安全性、包管理工具。

```dockerfile
# ❌ 错误：使用过大的镜像
FROM python:3.14.5

# ✅ 正确：使用 slim 变体减小体积
FROM python:3.14.5-slim

# 对于生产环境可考虑更小的 distroless
FROM gcr.io/distroless/python3-debian12
# 注意：distroless 不包含 shell、包管理器，调试困难，适合生产
```

| 基础镜像 | 大小 | 用途 | 含 shell | 包管理器 |
|----------|------|------|----------|----------|
| `python:3.14.5` | ~900MB | 开发/CI | 是 | apt |
| `python:3.14.5-slim` | ~120MB | 通用生产 | 是 | apt |
| `python:3.14.5-alpine` | ~50MB | 极致体积 | 是 | apk |
| `gcr.io/distroless/python3` | ~60MB | 安全优先 | 否 | 无 |

💡 **提示**：Alpine 使用 musl libc 而非 glibc，某些 Python 包（如 psycopg2）需要额外编译，推荐从 slim 镜像开始。

### 3.3 RUN 指令与层缓存

每个 `RUN`、`COPY`、`ADD` 指令都会创建一个新的镜像层。Docker 会缓存这些层，只有文件变化时才会重新构建。

```dockerfile
# ❌ 错误：多个 RUN 导致层数过多且无法利用缓存
FROM python:3.14.5-slim
RUN apt-get update
RUN apt-get install -y gcc
RUN pip install --no-cache-dir poetry
RUN pip install --no-cache-dir fastapi

# ✅ 正确：合并 RUN 并清理缓存
FROM python:3.14.5-slim
RUN apt-get update && \
    apt-get install -y --no-install-recommends gcc && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

# 利用缓存：先复制依赖文件，再安装依赖
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 最后复制源码（源码变化频率最高，放在最后）
COPY . .
```

### 3.4 COPY / ADD 区别

| 指令 | 特性 | 推荐场景 |
|------|------|----------|
| `COPY` | 仅复制本地文件到镜像 | 几乎所有场景 |
| `ADD` | 支持 URL 下载、自动解压 tar | 极少场景（有安全风险） |

```dockerfile
# ✅ 推荐：使用 COPY
COPY requirements.txt .
COPY src/ /app/src/

# ❌ 避免：使用 ADD 下载远程文件（无法利用层缓存，且不透明）
ADD https://example.com/app.tar.gz /tmp/

# ✅ ADD 唯一合理场景：自动解压本地 tar（但 COPY + RUN tar 更透明）
ADD app.tar.gz /opt/
```

💡 **提示**：官方 Docker 最佳实践推荐**优先使用 COPY**，ADD 只在需要自动解压 tar 时使用。URL 下载应使用 RUN curl/wget，这样缓存行为更可预测。

### 3.5 CMD / ENTRYPOINT

`CMD` 提供默认启动命令，`ENTRYPOINT` 定义容器的主命令。

```dockerfile
# CMD 可以被 docker run 后的参数覆盖
FROM python:3.14.5-slim
CMD ["python", "--version"]
# docker run myimage        → python --version
# docker run myimage whoami  → whoami（CMD 被覆盖）

# ENTRYPOINT 不能被覆盖（除非 --entrypoint）
FROM python:3.14.5-slim
ENTRYPOINT ["python"]
CMD ["--version"]
# docker run myimage           → python --version
# docker run myimage -c "..."  → python -c "..."
```

💡 **提示**：最佳实践是 `ENTRYPOINT` 放程序主命令，`CMD` 放默认参数。这样容器可以像使用命令行一样灵活。

### 3.6 构建上下文

```bash
# 构建上下文是 .（当前目录），Dockerfile 位于当前目录
docker build -t myapp:latest .

# 指定 Dockerfile 路径（但构建上下文仍为当前目录）
docker build -f docker/Dockerfile -t myapp:latest .
```

构建上下文中的所有文件和目录会被发送到 Docker daemon。如果项目庞大，务必使用 `.dockerignore` 避免发送不必要的文件。

### 3.7 .dockerignore

```dockerfile
# .dockerignore
.git
.gitignore
.vscode
__pycache__
*.pyc
.env
.venv
venv
*.md
tests/
.gitkeep
.terraform
node_modules
*.log
```

💡 **提示**：不写 `.dockerignore` 会导致构建上下文过大，构建速度慢。如果 `.dockerignore` 漏了 `.git`，会将整个 git 历史发送给 Docker daemon。

### 3.8 层缓存机制

```dockerfile
# ❌ 错误：先复制源码再安装依赖（每次源码变化都重新安装依赖）
FROM python:3.14.5-slim
COPY . .
RUN pip install --no-cache-dir -r requirements.txt

# ✅ 正确：先复制 requirements.txt，安装依赖，再复制源码
FROM python:3.14.5-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
```

**原理**：Docker 对每一层计算哈希。如果 `COPY` 的文件没有变化，这一层直接使用缓存。将变化频率最低的操作放在 Dockerfile 前面，变化最频繁的源码复制放在最后。

### 3.9 完整示例

```dockerfile
# Dockerfile（开发环境）
FROM python:3.14.5-slim

WORKDIR /app

# 系统依赖：放在最前，几乎不变
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
    curl \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*

# Python 依赖：依赖文件变化频率低
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 应用源码：变化最频繁
COPY . .

# 声明端口
EXPOSE 8000

# 默认启动命令
CMD ["uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**构建与运行：**

```bash
docker build -t demo-app:v1 .
docker run -d -p 8000:8000 --name demo demo-app:v1
curl http://localhost:8000
docker stop demo && docker rm demo
```

## 4 Dockerfile 进阶

### 4.1 为什么需要多阶段构建？

Python 应用在构建阶段需要 gcc、编译工具、构建依赖，但运行时只需要编译后的产物和运行时依赖。多阶段构建允许在同一个 Dockerfile 中使用多个 `FROM`，将构建产物复制到精简的运行镜像中。

### 4.2 多阶段构建

```dockerfile
# === 阶段 1：构建阶段 ===
FROM python:3.14.5-slim AS builder

WORKDIR /app

# 安装构建依赖
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
    gcc \
    libpq-dev \
    && rm -rf /var/lib/apt/lists/*

# 复制依赖文件
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt

# 复制源码
COPY . .

# === 阶段 2：运行阶段 ===
FROM python:3.14.5-slim

WORKDIR /app

# 从 builder 阶段复制 Python 包
COPY --from=builder /root/.local /root/.local

# 复制应用源码
COPY --from=builder /app /app

# 确保 PATH 包含用户安装的包
ENV PATH=/root/.local/bin:$PATH

# 创建非 root 用户
RUN addgroup --system app && \
    adduser --system --ingroup app appuser

USER appuser

HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
    CMD curl --fail http://localhost:8000/health || exit 1

EXPOSE 8000

CMD ["uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**构建与验证：**

```bash
docker build -t demo-app:multi-stage .
docker images demo-app
# 对比只使用单阶段的镜像大小
docker history demo-app:multi-stage
```

💡 **提示**：多阶段构建不会自动减小镜像体积——关键是要在最后阶段使用精简的基础镜像，并且只复制必要的产物。

### 4.3 多阶段构建对比

| 方法 | 镜像大小 | 构建速度 | 安全性 |
|------|---------|---------|--------|
| 单阶段（完整镜像） | ~900MB | 快（单次构建） | 低（含编译工具） |
| 单阶段（slim） | ~120MB | 中等 | 中等 |
| 多阶段（builder + slim） | ~130MB | 较慢（两次安装） | 高（不含编译工具） |
| 多阶段（builder + distroless） | ~70MB | 较慢 | 最高 |

### 4.4 ARG vs ENV

`ARG` 是构建时的变量，构建完成后消失；`ENV` 是运行时环境变量，在容器中持续存在。

```dockerfile
# ARG：构建时参数，在最终镜像中不可见（除非通过 ENV 保存）
ARG PYTHON_VERSION=3.14.5
FROM python:${PYTHON_VERSION}-slim

ARG APP_VERSION=1.0.0
ARG BUILD_DATE
LABEL version=${APP_VERSION}
LABEL build-date=${BUILD_DATE}

# ENV：运行时环境变量，在镜像中持续存在
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1
ENV APP_ENV=production

# 将 ARG 转为 ENV（如果需要运行时访问）
ARG APP_SECRET
ENV APP_SECRET=${APP_SECRET}
```

**构建时传参：**

```bash
docker build \
  --build-arg APP_VERSION=2.0.0 \
  --build-arg BUILD_DATE=$(date -u +'%Y-%m-%dT%H:%M:%SZ') \
  -t myapp:2.0.0 .
```

💡 **提示**：敏感信息（如 API 密钥）不应通过 ARG 传入，因为 `docker history` 会暴露它们。使用 `docker secrets` 或运行时环境变量。

### 4.5 ENTRYPOINT vs CMD 交互规则

| 组合 | ENTRYPOINT | CMD | `docker run` 带参数 | 行为 |
|------|-----------|-----|-------------------|------|
| 只有 CMD | - | `python app.py` | `bash` | 覆盖 CMD，执行 `bash` |
| 只有 ENTRYPOINT | `python` | - | `app.py` | 追加到 ENTRYPOINT：`python app.py` |
| 两者结合 | `python` | `app.py` | `--help` | 追加：`python --help` |
| Exec vs Shell | `["python"]` | `["app.py"]` | - | 不会创建 shell 进程 |

```dockerfile
# ❌ 错误：shell 格式（会通过 /bin/sh -c 运行，不会接收信号）
ENTRYPOINT python app.py

# ✅ 正确：exec 格式（直接运行，能接收 SIGTERM 信号）
ENTRYPOINT ["python", "app.py"]

# ❌ 错误：shell 格式
CMD python app.py

# ✅ 正确：exec 格式
CMD ["python", "app.py"]
```

💡 **提示**：**始终使用 exec 格式**（JSON 数组形式）。Shell 格式启动的进程是 `/bin/sh -c` 的子进程，不会接收 Docker 发送的 SIGTERM 信号，导致 `docker stop` 无法优雅关闭。

### 4.6 HEALTHCHECK

```dockerfile
# 基础健康检查
HEALTHCHECK CMD curl --fail http://localhost:8000/health || exit 1

# 完整配置
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
    CMD curl --fail --silent http://localhost:8000/health || exit 1
```

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--interval` | 30s | 健康检查间隔 |
| `--timeout` | 30s | 每次检查超时时间 |
| `--start-period` | 0s | 容器启动后延迟开始检查（应用启动时间） |
| `--retries` | 3 | 连续失败次数后标记为 unhealthy |

### 4.7 ONBUILD

`ONBUILD` 定义触发器，当当前镜像作为其他 Dockerfile 的基础镜像时触发。

```dockerfile
# 基础镜像 Dockerfile（供其他项目继承）
FROM python:3.14.5-slim

ONBUILD COPY requirements.txt .
ONBUILD RUN pip install --no-cache-dir -r requirements.txt
ONBUILD COPY . .
```

```dockerfile
# 子项目 Dockerfile
FROM my-base-image:latest
# 上面的 ONBUILD 指令会自动执行
CMD ["python", "app.py"]
```

💡 **提示**：`ONBUILD` 让基础镜像作者可以强制子项目执行某些操作，但缺点是行为隐式，子项目维护者可能不清楚发生了什么。

### 4.8 LABEL / EXPOSE

```dockerfile
# LABEL：添加元数据
LABEL org.opencontainers.image.title="FastAPI Demo"
LABEL org.opencontainers.image.description="Demo application for Docker learning"
LABEL org.opencontainers.image.version="1.0.0"
LABEL org.opencontainers.image.created="2026-06-02"
LABEL org.opencontainers.image.authors="developer@example.com"
LABEL org.opencontainers.image.source="https://github.com/example/docker-demo"
LABEL org.opencontainers.image.licenses="MIT"
```

## 5 镜像管理

### 5.1 为什么需要了解镜像管理？

构建完镜像后，需要分发给其他开发者或部署到生产服务器。理解镜像的存储、标记、推送和拉取机制是日常工作中不可或缺的技能。

### 5.2 docker tag / push / pull

```bash
# 标记镜像
docker tag demo-app:latest docker.io/yourusername/demo-app:1.0.0
docker tag demo-app:latest docker.io/yourusername/demo-app:latest

# 登录 Docker Hub
docker login
# 输入用户名和密码（推荐使用 Access Token，而非密码）

# 推送镜像
docker push docker.io/yourusername/demo-app:1.0.0
docker push docker.io/yourusername/demo-app:latest

# 拉取镜像
docker pull docker.io/yourusername/demo-app:1.0.0
docker pull nginx:latest
```

💡 **提示**：Docker Hub 公开镜像仓库免费，私有仓库需付费。替代方案有 GitHub Container Registry (`ghcr.io`) 和 Amazon ECR。

### 5.3 镜像分层与存储驱动

```bash
# 查看镜像的分层历史

# 查看镜像详细信息
docker inspect nginx:latest

# 查看各个层的大小
docker history --no-trunc nginx:latest
```

**存储驱动对比：**

| 驱动 | 性能 | 稳定性 | 适用平台 |
|------|------|--------|----------|

| fuse-overlayfs | 中等 | 高 | rootless Docker |
| devicemapper | 差 | 中 | 老旧系统（不推荐） |
| aufs | 中等 | 高 | 旧版 Ubuntu（已弃用） |

```bash
# 查看当前存储驱动

# 推荐输出：Storage Driver: overlay2
```

### 5.4 docker image history / inspect

### 5.6 镜像清理

```bash
# 列出所有镜像
docker images

# 删除特定镜像
docker rmi myapp:old-version

# 删除悬空镜像（没有标签的镜像）
docker image prune

# 删除所有未使用的镜像（不只是悬空）
docker image prune -a

# 清理所有未使用的资源（容器、网络、镜像、构建缓存）
docker system prune -a --volumes

# 按条件筛选删除（保留最近 N 个版本）
docker images --filter "reference=myapp:*" --format "{{.ID}}" | tail -n +6 | xargs docker rmi
```

## 6 容器网络

### 6.1 为什么需要了解容器网络？

容器之间需要相互通信（如应用连接数据库），也需要对外提供服务。Docker 提供多种网络模式，让容器间可以隔离或联通。

### 6.2 网络驱动

| 驱动 | 隔离性 | 通信范围 | 适用场景 |
|------|--------|----------|----------|
| bridge | 每个容器独立 IP | 同一宿主机 | 单机多容器通信（默认） |
| host | 无隔离，共享宿主机网络 | 宿主机 | 性能敏感，不需隔离 |
| overlay | 跨宿主机通信 | 多台机器 | Docker Swarm / K8s |
| none | 完全隔离 | 无 | 不需要网络的容器 |
| macvlan | 分配 MAC 地址 | 物理网络 | 需要直接接入物理网络 |

### 6.3 默认 bridge 网络

```bash
# 查看网络列表
docker network ls
# NETWORK ID     NAME      DRIVER    SCOPE
# abc123...      bridge    bridge    local
# def456...      host      host      local
# ghi789...      none      null      local

# 在默认 bridge 上运行容器
docker run -d --name web nginx:latest

# 查看容器的 IP
docker inspect web | grep IPAddress
# "IPAddress": "172.17.0.2"

# 默认 bridge 上容器间通过 IP 通信（不支持 DNS 名称）
```

### 6.4 用户自定义 bridge（推荐）

```bash
# 创建自定义 bridge 网络
docker network create --driver bridge app-network

# 指定子网和网关
docker network create \
  --driver bridge \
  --subnet=192.168.100.0/24 \
  --gateway=192.168.100.1 \
  app-network

# 在自定义网络上运行容器
docker run -d --name web --network app-network nginx:latest
docker run -d --name api --network app-network python:3.14.5-slim sleep infinity

# 自定义 bridge 支持 DNS 解析：
docker exec api ping web
# ✅ 可以解析！自定义 bridge 自动提供 DNS 解析

# 将已有容器连接到网络
docker network connect app-network existing-container

# 断开连接
docker network disconnect app-network existing-container
```

💡 **提示**：**始终使用自定义 bridge 网络**而非默认 bridge。自定义 bridge 提供：
1. 自动 DNS 解析（通过容器名访问）
2. 更好的隔离性
3. 可动态连接/断开容器

### 6.5 端口映射

```bash
# -p：指定端口映射
docker run -d -p 8080:80 nginx:latest
# 宿主机 8080 → 容器 80

# -P：随机映射所有暴露端口
docker run -d -P nginx:latest
docker ps
# PORTS: 0.0.0.0:32768->80/tcp

# 指定 IP 绑定
docker run -d -p 127.0.0.1:8080:80 nginx:latest
# 只能通过 localhost:8080 访问

# UDP 端口映射
docker run -d -p 5353:53/udp your-dns-image

# 查看端口映射
docker port container-name
```

### 6.6 host 网络模式

```bash
# host 模式：容器直接使用宿主机网络，不隔离
docker run -d --network host nginx:latest
# 无需 -p，直接通过宿主机 IP 访问

# 检查是否真的没有网络隔离
docker inspect --format='{{.HostConfig.NetworkMode}}' container-name
# 输出: host
```

⚠️ **提示**：host 网络模式性能最好，但安全性最差，仅建议在性能关键且可信任的环境中使用。

### 6.7 网络隔离示例

```bash
# 创建两个隔离的网络
docker network create frontend-net
docker network create backend-net

# 前端容器：可访问互联网和 backend-net，但无法访问 db-net
docker run -d --name nginx \
  --network frontend-net \
  nginx:latest

# 后端容器：同时连接 frontend-net 和 backend-net（作为网关）
docker run -d --name api \
  --network frontend-net \
  --network backend-net \
  your-api-image

# 数据库容器：仅在 backend-net 中，完全隔离
docker run -d --name postgres \
  --network backend-net \
  -e POSTGRES_PASSWORD=secret \
  postgres:16
```

**验证隔离：**

```bash
# API 可以 ping 通两者
docker exec api ping postgres   # ✅ 通
docker exec api ping nginx      # ✅ 通

# Nginx 无法 ping 通 postgres
docker exec nginx ping postgres # ❌ 不通
```

## 7 数据管理

### 7.1 为什么需要数据管理？

容器是临时的 —— `docker rm` 后容器内的所有数据都会丢失。数据库文件、上传的文件、日志等需要持久化存储，即使容器删除也能保留。

### 7.2 数据卷 vs 绑定挂载 vs tmpfs

| 特性 | 数据卷（Volume） | 绑定挂载（Bind Mount） | tmpfs |
|------|-----------------|----------------------|-------|
| 存储位置 | Docker 管理 `/var/lib/docker/volumes/` | 宿主机关任意路径 | 内存中 |
| 管理方式 | `docker volume` 命令管理 | 文件系统路径 | 无持久化 |
| 备份 | 简单（`docker run --volumes-from`） | 直接复制文件 | 不需要 |
| 多容器共享 | ✅ 支持 | ✅ 支持 | ❌ |
| 性能 | 好（原生） | 好（原生） | 最快 |
| 适用场景 | 数据库、持久化数据 | 开发时热重载、配置文件 | 临时敏感数据 |

### 7.3 数据卷（Volumes）

```bash
# 创建命名卷
docker volume create postgres-data

# 查看卷列表
docker volume ls

# 查看卷详细信息
docker volume inspect postgres-data

# 使用卷
docker run -d \
  --name postgres \
  -v postgres-data:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=secret \
  postgres:16

# 删除卷（必须没有容器使用）
docker volume rm postgres-data

# 删除所有未使用的卷
docker volume prune
```

**匿名卷：**

```bash
# 不指定名字，Docker 自动生成 UUID
docker run -d -v /var/lib/postgresql/data postgres:16
docker volume ls
# 输出类似：abc123456789...
```

### 7.4 绑定挂载（Bind Mounts）

```bash
# 将当前目录挂载到容器 /app
docker run -d \
  --name dev-app \
  -v "$(pwd):/app" \
  python:3.14.5-slim \
  python /app/app.py

# macOS/Linux 只读挂载
docker run -d --name dev-app \
  -v "$(pwd):/app:ro" \
  python:3.14.5-slim \
  python /app/app.py

# 挂载单个文件（如配置文件）
docker run -d --name app \
  -v "$(pwd)/config.yaml:/app/config.yaml:ro" \
  myapp:latest
```

💡 **提示**：绑定挂载是开发时最常用的方式——源码在宿主机修改，容器内自动生效（配合热重载工具如 uvicorn 的 `--reload`）。

### 7.5 数据卷容器模式

```bash
# 创建只共享数据卷的容器
docker create --name data-container \
  -v shared-data:/data \
  busybox

# 其他容器使用 --volumes-from 继承数据卷
docker run -d --name app1 --volumes-from data-container myapp
docker run -d --name app2 --volumes-from data-container myapp

# 数据卷容器不需要运行
docker ps -a | grep data-container
# 状态应为 Created（非运行，但数据卷仍然可用）
```

💡 **提示**：`--volumes-from` 是传统模式，现代 Docker Compose 中直接声明 `volumes` 即可，更清晰。

### 7.6 备份与恢复

```bash
# 备份：将卷内容导出到 tar 文件
docker run --rm \
  -v postgres-data:/data \
  -v "$(pwd):/backup" \
  busybox \
  tar czf /backup/postgres-backup-$(date +%Y%m%d).tar.gz -C /data .

# 恢复：将 tar 文件导入卷
docker run --rm \
  -v postgres-data:/data \
  -v "$(pwd):/backup" \
  busybox \
  tar xzf /backup/postgres-backup-20260602.tar.gz -C /data

# 验证备份文件
ls -la postgres-backup-*.tar.gz
```

**数据库专用备份：**

```bash
# PostgreSQL 使用 pg_dump
docker exec -t postgres pg_dump -U myuser mydb > backup.sql

# 恢复
cat backup.sql | docker exec -i postgres psql -U myuser mydb
```

### 7.7 挂载方式对比总结

```bash
# Volume：Docker 管理
docker run -v myvolume:/app/data myapp

# Bind mount：用户指定路径
docker run -v /host/path:/container/path myapp

# tmpfs：内存中
docker run --tmpfs /app/tmp:noexec,nosuid,size=64m myapp
```

```dockerfile
# Dockerfile 中声明的 VOLUME 会自动创建匿名卷
VOLUME /app/data
# 作用：防止运行时忘记挂载导致数据丢失
# 限制：无法指定路径，无法用于绑定挂载提示
```

## 8 Docker Compose 基础

### 8.1 为什么需要 Docker Compose？

一个应用通常由多个服务组成（Web 服务器、数据库、缓存等）。如果每次都要手动运行多个 `docker run` 命令，不仅繁琐还容易出错。Docker Compose 用声明式 YAML 文件定义并运行多容器应用。

### 8.2 docker-compose.yml 结构

```yaml
# docker-compose.yml
services:
  web:
    build: .
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=postgresql+asyncpg://user:pass@db:5432/mydb
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

**启动与停止：**

```bash
# 启动所有服务
docker compose up -d

# 查看运行状态
docker compose ps

# 查看日志
docker compose logs -f

# 停止所有服务
docker compose down

# 停止并删除卷（⚠️ 数据会丢失）
docker compose down -v
```

### 8.3 顶级元素

```yaml
services:
  # 定义应用的服务（容器）

networks:
  # 定义网络（默认会自动创建 project-name_default）

volumes:
  # 定义数据卷

configs:
  # 定义配置（Docker Swarm）

secrets:
  # 定义机密信息
```

### 8.4 常用命令

```bash
# 构建并启动（-d 后台运行）
docker compose up -d

# 停止并删除容器、网络，但不删除卷
docker compose down

# 停止并删除所有（包括卷）
docker compose down -v

# 列出服务的容器
docker compose ps

# 查看日志（-f 持续跟踪，--tail 指定行数）
docker compose logs -f web
docker compose logs --tail=100 db

# 进入运行中的容器
docker compose exec web bash

# 执行单条命令
docker compose exec db pg_isready -U myuser

# 重建某个服务（修改 Dockerfile 后）
docker compose up -d --build web

# 查看当前配置
docker compose config

# 拉取所有镜像
docker compose pull
```

### 8.5 端口映射

```yaml
services:
  web:
    ports:
      # 完整语法：HOST:CONTAINER
      - "8000:8000"
      # 短语法：只指定容器端口（随机分配宿主机端口）
      - "8000"
      # 指定协议
      - "8000:8000/tcp"
      - "5353:53/udp"
      # 绑定到特定 IP
      - "127.0.0.1:8000:8000"

  db:
    # 数据库通常不对外暴露，只在内部网络通信
    ports:
      - "5432"  # 仅内部使用，不暴露给宿主机
```

### 8.6 卷在 Compose 中

```yaml
services:
  db:
    image: postgres:16
    volumes:
      # 命名卷（推荐）
      - pgdata:/var/lib/postgresql/data
      # 绑定挂载
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql:ro
      # 匿名卷
      - /var/lib/postgresql/data

  app:
    build: .
    volumes:
      # 开发时绑定挂载源码
      - ./src:/app/src:ro

volumes:
  pgdata:       # 声明命名卷
  cache-data:   # 另一个命名卷
```

## 9 Docker Compose 进阶

### 9.1 为什么需要 Compose 进阶功能？

基础 Compose 定义足以运行简单应用。但在生产环境中，需要健康检查保障服务的可用性、条件启动顺序、多环境配置等进阶能力。

### 9.2 healthcheck 配置

```yaml
services:
  app:
    build: .
    healthcheck:
      test: ["CMD", "curl", "--fail", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 15s  # 启动后的缓冲期

  db:
    image: postgres:16
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U myuser -d mydb"]
      interval: 10s
      timeout: 5s
      retries: 5
```

**查看健康状态：**

```bash
docker compose ps
# 显示 Status 列：healthy / unhealthy / starting

docker inspect --format='{{.State.Health.Status}}' <container-name>
```

### 9.3 depends_on 条件控制

```yaml
services:
  web:
    build: .
    depends_on:
      db:
        condition: service_healthy   # 等待 db 健康检查通过
      redis:
        condition: service_started   # 只需启动（默认行为）
      cache:
        condition: service_completed_successfully  # 某一次性服务完成后

  db:
    image: postgres:16
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U myuser"]
      interval: 5s
      timeout: 5s
      retries: 5

  cache:
    image: busybox
    command: ["/bin/sh", "-c", "echo Warming up... && sleep 5"]
```

💡 **提示**：`depends_on` 只控制启动顺序，不保证内部服务就绪。使用 `condition: service_healthy` + `healthcheck` 才能真正保障服务可用。

### 9.4 profiles 可选服务

```yaml
services:
  # 始终启动
  app:
    build: .
    ports:
      - "8000:8000"

  # 仅开发环境
  adminer:
    image: adminer
    ports:
      - "8080:8080"
    profiles:
      - dev
      - staging

  # 仅调试用
  pgadmin:
    image: dpage/pgadmin4
    profiles:
      - debug

  # 仅生产环境
  sentry:
    image: sentry:latest
    profiles:
      - production
```

```bash
# 仅启动核心服务
docker compose up -d

# 启动核心 + dev profile 服务
docker compose --profile dev up -d

# 启动多个 profile
docker compose --profile dev --profile debug up -d

# 生产环境
docker compose --profile production up -d
```

### 9.5 多文件合并

```yaml
# docker-compose.yml（基础配置）
services:
  app:
    build: .
    ports:
      - "8000:8000"
    environment:
      - APP_ENV=development

  db:
    image: postgres:16
```

```yaml
# docker-compose.prod.yml（生产覆盖）
services:
  app:
    ports:
      - "80:8000"
    environment:
      - APP_ENV=production
    deploy:
      replicas: 3

  db:
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
```

```bash
# 开发环境
docker compose up -d

# 生产环境（合并基础配置和生产覆盖）
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d

# 验证合并后的配置
docker compose -f docker-compose.yml -f docker-compose.prod.yml config
```

### 9.6 extends / 扩展字段

```yaml
# docker-compose.yml
x-logging: &default-logging
  driver: "json-file"
  options:
    max-size: "10m"
    max-file: "3"

x-app-common: &app-common
  restart: unless-stopped
  logging:
    <<: *default-logging
  networks:
    - app-net

services:
  web:
    <<: *app-common
    build: .
    ports:
      - "8000:8000"

  worker:
    <<: *app-common
    build: .
    command: celery -A tasks worker

networks:
  app-net:
    driver: bridge
```

💡 **提示**：YAML 锚点 `&` 和引用 `<<: *` 可以有效消除 Compose 文件中的重复配置，但不要过度使用以免降低可读性。

## 10 环境与配置

### 10.1 为什么需要关注配置管理？

应用配置（数据库连接、密钥、环境标识等）在不同环境（开发、测试、生产）间差异很大。将配置硬编码在 Dockerfile 或代码中会带来安全风险和维护灾难。

### 10.2 .env 文件

```bash
# .env（注意文件名是 .env 而不是 .env.file）
# Docker Compose 自动加载同级目录下的 .env 文件

# 项目信息
PROJECT_NAME=docker-demo
APP_VERSION=1.0.0

# 数据库
POSTGRES_USER=myuser
POSTGRES_PASSWORD=changeme_in_production
POSTGRES_DB=mydb
POSTGRES_HOST=db
POSTGRES_PORT=5432

# Redis
REDIS_HOST=redis
REDIS_PORT=6379

# 应用
APP_ENV=development
DEBUG=true
SECRET_KEY=dev-secret-key-not-for-production
```

```yaml
# docker-compose.yml 中使用 .env 变量
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: ${POSTGRES_USER:-myuser}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-changeme}
      POSTGRES_DB: ${POSTGRES_DB:-mydb}

  app:
    build: .
    environment:
```

```python
import os

    secret_path = Path(f"/run/secrets/{secret_name}")
```

**FastAPI 集成 Prometheus：**

```python
# src/metrics.py
import time

from fastapi import Request, Response
from prometheus_client import Counter, Histogram, generate_latest, CONTENT_TYPE_LATEST
from starlette.middleware.base import BaseHTTPMiddleware

REQUEST_COUNT = Counter(
    "http_requests_total",
    "Total HTTP requests",
    ["method", "endpoint", "status"],
)

REQUEST_DURATION = Histogram(
    "http_request_duration_seconds",
    "HTTP request duration in seconds",
    ["method", "endpoint"],
)


class MetricsMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        start_time = time.monotonic()
        response: Response = await call_next(request)
        duration = time.monotonic() - start_time

        REQUEST_COUNT.labels(
            method=request.method,
            endpoint=request.url.path,
            status=response.status_code,
        ).inc()

        REQUEST_DURATION.labels(
            method=request.method,
            endpoint=request.url.path,
        ).observe(duration)

        return response
```

## 12 安全

### 12.1 为什么需要关注容器安全？

容器共享宿主机内核，配置不当可能导致容器逃逸、权限提升、数据泄露。安全性是生产环境部署的底线要求。

### 12.2 非 root 用户运行

⚠️ **默认情况下，容器以 root 用户运行**。如果攻击者突破应用层能获得容器内的 root 权限，可以直接操作宿主机。

❌ **错误：以 root 运行**

```dockerfile
FROM python:3.14.5-slim
COPY . /app
WORKDIR /app
RUN pip install -r requirements.txt
CMD ["python", "app.py"]
# 容器内进程以 root 运行，存在安全隐患
```

✅ **正确：创建非 root 用户**

```dockerfile
FROM python:3.14.5-slim

RUN addgroup --system --gid 1001 appgroup && \
    adduser --system --uid 1001 --ingroup appgroup --no-create-home appuser

WORKDIR /app

COPY --chown=appuser:appgroup requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY --chown=appuser:appgroup . .

USER appuser

HEALTHCHECK CMD curl --fail http://localhost:8000/health || exit 1

EXPOSE 8000

CMD ["uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**验证用户：**

```bash
docker exec myapp whoami
# appuser

docker exec myapp id
# uid=1001(appuser) gid=1001(appgroup)
```

### 12.3 只读文件系统

```yaml
# docker-compose.yml
services:
  app:
    build: .
    read_only: true
    tmpfs:
      - /tmp
```

**在运行时限制：**


💡 **提示**：只读文件系统可以有效阻止攻击者写入恶意文件。但要注意应用可能需要在 `/tmp` 写入临时文件，通过 `--tmpfs` 解决。

## 14 性能与资源管理

### 14.1 为什么需要关注性能？

不合理的资源分配会导致容器争抢宿主机资源、应用响应缓慢甚至 OOM（内存溢出）。理解 Docker 的资源管理机制能让应用运行得更稳定高效。

### 14.2 CPU / 内存限制

```bash
# CPU 限制
docker run -d \
  --cpus="1.5" \           # 最多使用 1.5 核
  --cpuset-cpus="0,1" \    # 指定使用 CPU 0 和 1
  --cpu-shares=1024 \      # 相对权重（默认 1024）
  myapp:latest

# 内存限制
docker run -d \
  --memory="256m" \             # 硬限制：最多 256MB
  --memory-reservation="128m" \ # 软限制：尽力保留 128MB
  --memory-swap="512m" \        # 内存+交换分区上限
  --oom-kill-disable=false \    # 内存超限时是否 OOM Kill
  myapp:latest
```

```yaml
# docker-compose.yml
services:
  app:
    build: .
    deploy:
      resources:
        limits:
          cpus: "0.5"
          memory: "256M"
        reservations:
          cpus: "0.25"
          memory: "128M"
```

### 14.4 存储驱动优化


### 14.6 并发控制

```python
# src/connection_pool.py
from contextlib import asynccontextmanager
from dataclasses import dataclass

from sqlalchemy.ext.asyncio import (
    AsyncSession,
    async_sessionmaker,
    create_async_engine,
)

from src.config import settings


@dataclass
class DatabaseConfig:
    """数据库连接池配置"""

    pool_size: int = 10        # 连接池大小
    max_overflow: int = 20     # 超出 pool_size 的最大连接数
    pool_timeout: int = 30     # 等待连接的超时时间
    pool_recycle: int = 1800   # 连接回收时间（秒）
    echo: bool = False         # SQL 日志


class DatabaseManager:
    """数据库连接池管理器"""

    def __init__(self, config: DatabaseConfig | None = None):
        self.config = config or DatabaseConfig()
        self._engine = None
        self._session_factory = None

    def init(self, database_url: str):
        self._engine = create_async_engine(
            database_url,
            pool_size=self.config.pool_size,
            max_overflow=self.config.max_overflow,
            pool_timeout=self.config.pool_timeout,
            pool_recycle=self.config.pool_recycle,
            echo=self.config.echo,
        )
        self._session_factory = async_sessionmaker(
            self._engine,
            class_=AsyncSession,
            expire_on_commit=False,
        )

    async def close(self):
        if self._engine:
            await self._engine.dispose()

    @asynccontextmanager
    async def get_session(self) -> AsyncSession:
        if not self._session_factory:
            raise RuntimeError("Database not initialized")
        async with self._session_factory() as session:
            try:
                yield session
                await session.commit()
            except Exception:
                await session.rollback()
                raise
            finally:
                await session.close()

    async def check_health(self) -> bool:
        if not self._engine:
            return False
        try:
            async with self._engine.connect() as conn:
                await conn.execute(DBText("SELECT 1"))
            return True
        except Exception:
            return False
```

### 15.5 优雅关闭

```python
# src/shutdown.py
import asyncio
import signal
import logging
from contextlib import asynccontextmanager

from fastapi import FastAPI

logger = logging.getLogger(__name__)


class GracefulShutdown:
    """优雅关闭管理器"""

    def __init__(self, app: FastAPI, timeout: int = 30):
        self.app = app
        self.timeout = timeout
        self._shutdown_event = asyncio.Event()

    async def wait_for_shutdown(self):
        """等待关闭信号"""
        loop = asyncio.get_event_loop()

        def _signal_handler():
            logger.info("Received shutdown signal, starting graceful shutdown...")
            self._shutdown_event.set()

        for sig in (signal.SIGTERM, signal.SIGINT):
            loop.add_signal_handler(sig, _signal_handler)

        logger.info(
            "Shutdown handler registered, waiting for signal (timeout=%ds)...",
            self.timeout,
        )
        await asyncio.wait_for(
            self._shutdown_event.wait(),
            timeout=self.timeout,
        )

    async def shutdown(self):
        """执行关闭清理"""
        logger.info("Closing database connections...")
        from src.database import db_manager
        await db_manager.close()

        logger.info("Closing Redis connections...")
        from src.cache import redis_client
        await redis_client.close()

        logger.info("Graceful shutdown complete")
```

```python
# src/main.py 中注册生命周期事件
from contextlib import asynccontextmanager

from fastapi import FastAPI

from src.database import db_manager
from src.config import settings
from src.health import router as health_router
from src.metrics import MetricsMiddleware
from src.sentry_setup import setup_sentry
from src.shutdown import GracefulShutdown


def create_app() -> FastAPI:
    """应用工厂"""

    shutdown_handler = GracefulShutdown(timeout=30)

    @asynccontextmanager
    async def lifespan(app: FastAPI):
        # 启动
        setup_sentry()
        db_manager.init(settings.database_url)
        logger.info("Application started")
        yield
        # 关闭
        await shutdown_handler.shutdown()
        logger.info("Application stopped")

    app = FastAPI(
        title=settings.project_name,
        version=settings.app_version,
        lifespan=lifespan,
    )

    app.add_middleware(MetricsMiddleware)
    app.include_router(health_router)

    return app


app = create_app()
```

## 17 CI/CD 集成

### 17.1 为什么需要 CI/CD？

手动构建、测试、部署应用不仅效率低下，而且容易出错。CI/CD 自动化了从代码提交到生产部署的整个流程，确保每次变更都经过构建、测试、安全扫描和部署。

### 17.2 GitHub Actions 流水线

```yaml
# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  lint-and-test:
    name: Lint and Test
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: myuser
          POSTGRES_PASSWORD: testpass
          POSTGRES_DB: testdb
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
      redis:
        image: redis:7-alpine
        ports:
          - 6379:6379
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.14"

      - name: Cache dependencies
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}
          restore-keys: |
            ${{ runner.os }}-pip-

      - name: Install dependencies
        run: |
          pip install --no-cache-dir -r requirements.txt
          pip install pytest httpx

      - name: Run linting
        run: |
          pip install ruff
          ruff check src/ tests/

      - name: Run tests
        env:
          DATABASE_URL: postgresql+asyncpg://myuser:testpass@localhost:5432/testdb
          REDIS_URL: redis://localhost:6379/0
          SECRET_KEY: test-secret-key
          APP_ENV: testing
        run: |
          pytest tests/ -v --cov=src --cov-report=xml

      - name: Upload coverage
        uses: codecov/codecov-action@v4
        with:
          file: ./coverage.xml

  docker-build-and-scan:
    name: Build and Scan Docker Image
    needs: lint-and-test
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
      security-events: write

    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=ref,event=branch
            type=sha,format=short
            type=raw,value=latest,enable=${{ github.ref == 'refs/heads/main' }}

      - name: Build and push
        uses: docker/build-push-action@v6
        with:
          context: .
          push: ${{ github.event_name != 'pull_request' }}
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

      - name: Scan Docker image
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          format: sarif
          output: trivy-results.sarif
          severity: CRITICAL,HIGH

      - name: Upload Trivy results
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: trivy-results.sarif

  deploy:
    name: Deploy to Production
    needs: [lint-and-test, docker-build-and-scan]
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'

    steps:
      - uses: actions/checkout@v4

      - name: Deploy to server
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            cd /opt/app
            docker compose pull
            docker compose up -d --remove-orphans
            docker system prune -af --filter "until=24h"
```

### 18.1 镜像瘦身原则

**最小化镜像大小的策略：**

```dockerfile
# 1. 选择合适的基础镜像
FROM python:3.14.5-slim  # ✅ 推荐，平衡大小和功能
# FROM python:3.14.5     # ❌ 太大，包含编译工具
# FROM alpine:3.20       # 太小，兼容性问题

# 2. 合并 RUN 指令并清理缓存
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
    curl \
    libpq-dev \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/* /tmp/* /var/tmp/*

# 3. 多阶段构建
FROM python:3.14.5-slim AS builder
# ... 构建步骤 ...
FROM python:3.14.5-slim
COPY --from=builder /usr/local/lib/python3.14/site-packages /usr/local/lib/python3.14/site-packages

# 4. 不要安装不必要的包
RUN pip install --no-cache-dir --only-binary=:all: -r requirements.txt

# 5. 使用 .dockerignore 排除构建上下文中的无关文件
```

| 策略 | 示例 | 压缩效果 |
|------|------|---------|
| 使用 slim 镜像 | slim 替代 full | -85% |
| 多阶段构建 | builder + runtime | -30% |
| 清理 APT 缓存 | `rm -rf /var/lib/apt/lists/*` | -20MB |
| 只安装运行时依赖 | --no-install-recommends | -50MB |
| Pip 无缓存 | --no-cache-dir | -30MB |

```dockerfile
# 确保 Dockerfile 中设置了正确的 WORKDIR

from src.config import settings  # 而非 from config import settings
```

```bash
# 解决：检查端口映射
docker run -p 8080:80 nginx:latest

# 问题：外部无法访问
```

### 19.7 常见错误对照表

| 错误信息 | 原因 | 解决方案 |
|---------|------|---------|
| `port is already allocated` | 端口被占用 | 换端口或停止占用进程 |
| `no matching manifest for linux/arm64` | 镜像不支持 ARM | 使用 `--platform linux/amd64` |
| `Cannot connect to the Docker daemon` | Docker 未运行 | 启动 Docker Desktop/Engine |
| `docker: 'compose' is not a docker command` | 未安装 Compose v2 | 安装 docker-compose-plugin |
| `Error response from daemon: conflict` | 容器名已存在 | 使用 `docker rm` 删除旧容器 |
| `OCI runtime exec failed` | 容器内无目标可执行文件 | 检查容器内命令（sh vs bash） |
| `no space left on device` | 磁盘满 | `docker system prune -a` |
| `unauthorized: access token refused` | Docker Hub 认证失败 | `docker login` 重新登录 |

### 20.5 快速参考命令

```bash
# 镜像相关
docker images                     # 列出镜像
docker pull <image>               # 拉取镜像
docker push <image>               # 推送镜像
docker rmi <image>                # 删除镜像
docker build -t <tag> .           # 构建镜像
docker tag <src> <target>         # 标记镜像
docker history <image>            # 查看镜像历史
docker inspect <image>            # 查看镜像详情

# 容器相关
docker run -d -p <host>:<container> <image>  # 运行容器
docker ps [-a]                    # 列出容器
docker stop <container>           # 停止容器
docker start <container>          # 启动容器
docker restart <container>        # 重启容器
docker rm [-f] <container>        # 删除容器
docker exec -it <container> bash  # 进入容器
docker logs [-f] <container>      # 查看日志
docker cp <src> <dest>            # 复制文件
docker top <container>            # 容器进程
docker stats                      # 资源统计

# 网络相关
docker network ls                 # 列出网络
docker network create <name>      # 创建网络
docker network rm <name>          # 删除网络
docker network connect <net> <container>  # 连接网络
docker network disconnect <net> <container>  # 断开网络

# 卷相关
docker volume ls                  # 列出卷
docker volume create <name>       # 创建卷
docker volume rm <name>           # 删除卷
docker volume prune               # 清理未使用的卷

# Compose 相关
docker compose up -d              # 启动服务
docker compose down               # 停止服务
docker compose ps                 # 列出服务
docker compose logs -f            # 查看日志
docker compose exec <service> bash  # 进入服务
docker compose build              # 构建镜像
docker compose pull               # 拉取镜像
docker compose config             # 验证配置

# 系统相关
docker system df                  # 磁盘使用
docker system prune [-a]          # 清理
docker info                       # 系统信息
docker version                    # 版本信息

# 安全相关
docker scout cves <image>         # 漏洞扫描
docker scout quickview <image>    # 快速概览
```
